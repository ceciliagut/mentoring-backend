# Session 08 — Domain events

## Goal

Understand that the domain can express important facts without knowing about Kafka or any infrastructure.

---

## Concepts covered

- What is a domain event
- Events generated by the aggregate
- When to emit an event
- The difference between state and event

---

## Key questions to discuss

- What is the difference between a state change and an event?
- Who produces domain events — the aggregate, the use case, or the controller?
- Should a domain event know about Kafka? About JSON serialization?
- How does the application layer know that an event was produced?
- What is the difference between a domain event and an integration event?

---

## Expected outcome

- Domain events defined: `ReservationRequested`, `ReservationConfirmed`, `ReservationCancelled`
- Aggregate returns events as part of its operations
- Application layer prepared to publish them (without Kafka yet)
- Unit tests covering event generation

---

## Exercise for the next two weeks

- Define domain events as simple data classes in `domain/event`
- Make aggregate operations return or collect events (e.g. return a list of events)
- In the use case, collect the events and log them for now
- Write unit tests asserting that the correct events are produced for each operation
- Bring a PR — domain events must have zero infrastructure dependencies

---

## Review questions

- Why should a domain event not contain a `@JsonProperty` annotation?
- What is the difference between `ReservationRequested` (domain event) and a Kafka message?
- How does the application layer decide what to do with a domain event?
- What problem does the outbox pattern solve? (Conceptual — we will implement it later.)
